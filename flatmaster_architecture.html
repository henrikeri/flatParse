1. Scope

FlatMaster is a desktop orchestration application for flat-frame calibration workflows. It scans flat and dark libraries, computes deterministic matching decisions, materializes required master darks, and runs final flat calibration/integration via either:

  • PixInsight script automation (PixInsightService + templates), or
  • Native in-process stacking (NativeProcessingService + ImageStackingAlgorithms).

Primary solution directories:

  • src/FlatMaster.Core (contracts + models)
  • src/FlatMaster.Infrastructure (service implementations)
  • src/FlatMaster.WPF (UI, orchestration ViewModel, app composition)
  • tests/FlatMaster.Tests (unit tests)

3. Runtime Flows

Scan Flow

  1. User triggers ScanAll.
  2. If flats and darks roots are on different drives, both scans run in parallel; otherwise sequentially.
  3. Flat scan returns grouped DirectoryJob objects with exposure groups and average flat temperature.
  4. Dark scan catalogs dark/master/bias files and associates each with nearest DARKS### ancestor when present.
  5. After both scans complete, matching is auto-generated.

Process Flow

  1. Build immutable ProcessingPlan from selected UI rows and overrides.
  2. Preview and optionally materialize missing master darks first.
  3. Run selected engine:
    • Native: in-process calibration + integration
    • PixInsight: per-folder script invocation, template-selected mode
  4. Regenerate matching diagnostics and produce processing report.

5. Dark Matching Subsystem

DarkMatchingService.FindBestDark implements a strict priority waterfall. Temperature gate and bias-guard thresholds are first-class controls.

Core rules

  1. P1: Master dark exact exposure.
  2. P2: Loose dark exact exposure.
  3. P3: Master dark within + or -2 s.
  4. P4: Loose dark within + or -2 s.
  5. P5: Master dark within + or -10 s (outside +/-2 s).
  6. P6: Loose dark within + or -10 s (outside +/-2 s).
  7. P7: Bias fallback (MasterBias first, then Bias).
  8. P8: No match.

Strict pass uses MaxTempDeltaC when both flat and dark temperatures exist. A second relaxed pass can still select a dark before bias if both absolute deltas are within configured guards:

  • DarkOverBiasTempDeltaC
  • DarkOverBiasExposureDeltaSeconds

Deterministic tie-breaking: score, then type priority, then file path lexical order.

6. Master Dark Materialization

Normal mode

  1. Analyze selected flat groups and collect folders requiring loose-dark integration.
  2. Treat all content under top-level DARKS### folder as one source bucket.
  3. Prefer reuse when valid master/manifest is present and key hash matches current inputs.
  4. Materialize via Native or PixInsight path.
  5. Register created masters back into plan-level catalog so subsequent groups reuse them.

Darks-only mode

  1. Cluster raw darks by exposure, binning, gain, and temperature buckets.
  2. Temperature clustering tolerance is passed from UI (default 1.0 degC).
  3. Output path convention: OutputRoot/Master/Darks/<seconds>s/<temp>degC/MasterDark_...<ext>

7. Processing Engines

PixInsight Engine

  • One-folder-per-invocation policy for stability (batchSize = 1 enforced).
  • Template selection:
    • PixInsightTemplate.pjsr for flat calibration + integration
    • PixInsightTemplate.DARKS.pjsr for dark integration only
  • Completion signaling via sentinel file and detached process detection.
  • No preflight stage in current implementation.

Native Engine

  • Multithreaded frame load/calibration and pixel stack integration.
  • Flat calibration uses dark subtraction only when a dark/bias is matched.
  • Rejection strategy by stack size:
    • <3: Average
    • 3-5: Percentile Clip
    • 6-15: Winsorized Sigma Clip
    • >15: Linear Fit Clipping
  • Writes either XISF or FITS based on config.

8. Concurrency and Performance

  • Metadata reader parallelism auto-tuned from CPU count and available memory.
  • Flat and dark scans can run concurrently when roots are on different drives.
  • Native processing uses bounded parallel loops and partitioned pixel ranges.
  • Master dark materialization queue executes in parallel for native mode and serially where engine constraints require it.

9. Persistence, Logging, and Diagnostics

UI settings persistence

Stored at %LOCALAPPDATA%/FlatMaster/user-settings.json.

Persists directory roots, output path, output format, checkboxes, threshold values, and selection state.

Logs

  • Session log path is shown in UI log at startup.
  • Detailed app log is written to temp with timestamped filename.
  • Processing progress messages are the canonical UI progress input.

Licensing files

License and third-party notices are embedded as resources and extracted on demand for self-contained deployments.

11. Failure Model and Recovery

  • All long-running operations are cancellation-aware.
  • Materialization aggregates per-folder errors and throws summarized exceptions.
  • PixInsight execution retries multiple argument variants when startup/runtime behavior is unstable.
  • UI remains orchestration authority: validation before execution, explicit user confirmation before heavy materialization.

12. Extension Points

  • New file formats: extend FitsImageIO and metadata reader.
  • Alternative matching heuristics: evolve IDarkMatchingService implementation while preserving DarkMatchResult.
  • Alternative engine: implement IImageProcessingEngine and plug via DI.
  • Additional report metrics: extend ProcessingReportService and model.

Contract stability recommendation: keep Core interfaces backward-compatible and place experimental behavior behind configuration toggles.
