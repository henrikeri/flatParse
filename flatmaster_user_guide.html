1. What FlatMaster Does

FlatMaster scans flat and dark libraries, matches each flat exposure group to the best dark/bias candidate, materializes missing master darks when required, and creates final master flats using either PixInsight or the native engine.

2. Requirements

Platform and Runtime

  • Windows 10/11 x64.
  • .NET Desktop Runtime 8.x installed on target machine.
  • Read access to input roots and write access to output root.

File Types

  • Supported image extensions: .fits, .fit, .xisf.
  • Unsupported extensions are ignored during scan.
  • Metadata can be read from headers or inferred from filename patterns.

PixInsight Requirements (only if PixInsight engine is selected)

  • Valid PixInsight.exe path in Configuration tab.
  • Automation mode enabled by installation defaults.

Native mode does not require PixInsight.

3. Quick Start Workflow

  1. Configuration tab: set engine, output root, output format, and roots.
  2. Scan tab: click Scan Flats + Darks (or darks-only scan in darks-only mode).
  3. Matching Details tab: click Run Matching, inspect rows, apply overrides if needed.
  4. Process and Report tab: click Start Processing.
  5. Review process log and final report.
Matching is manual by design. Re-scanning clears previous matching results and requires a new Run Matching pass.

4. Feature Overview

Scan Features

  • Separate flat and dark progress bars.
  • Directory/file counters while scanning.
  • Abort support with cancellation-aware loops.

Matching Features

  • Group-level diagnostics with confidence and temp delta.
  • Expandable rows to inspect file members.
  • Override list supports existing masters and loose-dark candidates.

Processing Features

  • PixInsight and native processing paths.
  • Darks-only mode for master dark generation.
  • Automatic pre-materialization of missing dark masters.

5. Folder Parsing Rules and Constraints

5.1 Global Traversal Rules

  • Traversal is recursive from each configured root.
  • Directories skipped by name (case-insensitive): _darkmasters, _calibratedflats, masters, _processed, and dot-prefixed directories.
  • Only top-level files in each visited directory are enumerated for parsing.

5.2 Flat Parsing Constraints

  • A file is eligible only when metadata has exposure time and image type is not dark/bias class.
  • Grouping key: exposure rounded to 3 decimals (for example 10.000s -> 10s style key).
  • Grouping scope: per directory and per exposure key.
  • Average flat temperature is computed from group members with valid temperature.

5.3 Dark Parsing Constraints

  • Dark catalog includes: Dark, DarkFlat, MasterDark, MasterDarkFlat, Bias, MasterBias.
  • Bias/MasterBias may have missing exposure; dark classes require exposure (unless repaired from manifest).
  • Master dark pre-scan looks for *MasterDark*.fits/.fit/.xisf.
  • MasterDark.meta.json can repair missing exposure/temperature/binning/gain.

5.4 DARKSXXX Root Constraint (Critical)

For loose dark materialization (priorities P2/P4/P6), frames are grouped strictly by the nearest ancestor folder matching: 

DARKSXXX or DARKXXX, where XXX is 2 to 4 digits (regex: ^DARKS?\d{2,4}$)
  • All deeper subfolders under that DARKSXXX root are treated as one group source.
  • If a loose dark is outside a DARKSXXX hierarchy, it is not used for auto-materialization.
Recommended: keep all loose dark libraries under properly named DARKSXXX roots.

5.5 Metadata and Filename Inference

When headers are missing values, FlatMaster attempts filename inference:

  • Exposure inference patterns include values like 10s and EXPOSURE10.00.
  • Temperature inference pattern includes tokens like temp_-10.0.
  • Image type inference uses filename/header keywords (MASTERDARK, DARK, FLAT, BIAS, etc.).
If both headers and filename lack exposure, that frame cannot participate in exposure-based dark matching.

6. Dark Matching Priorities

Matching order is deterministic:

PriorityCandidate TypeExposure Rule
P1MasterDark / MasterDarkFlatExact exposure
P2Dark / DarkFlat (loose)Exact exposure
P3MasterDark / MasterDarkFlatNearest within 2s
P4Dark / DarkFlat (loose)Nearest within 2s
P5MasterDark / MasterDarkFlatNearest within 10s (outside 2s)
P6Dark / DarkFlat (loose)Nearest within 10s (outside 2s)
P7MasterBias / BiasFallback
P8No matchNo usable dark/bias candidate

Additional gate: dark-over-bias thresholds (temperature and exposure) can allow a non-strict dark to beat bias fallback when within limits.

7. Master Dark Materialization Rules

Main Flat Workflow

  • If selected match is loose dark (P2/P4/P6), FlatMaster creates a master dark first.
  • Created master is reused across all applicable flat groups.
  • Materialization happens before flat calibration/integration starts.

Darks-Only Workflow

  • No flat processing.
  • Raw darks bucketed by exposure/binning/gain and +/-1.0 degC tolerance.
  • One master produced per resulting bucket.

Important Constraints

  • Darks are integrated directly (not calibrated as flats).
  • Rows with fewer than 3 flat frames are marked warning and default to unchecked in Matching Details.
  • Unchecked rows remain visible and are excluded only from processing selection.

8. Output Structure and Naming

Master Darks

OutputRoot\Master\Darks\25s\-10degC\MasterDark_25s_-10degC.xisf
OutputRoot\Master\Darks\25s\-10degC\MasterDark_25s_-10degC.fits
  • No GUID segment in final folder path.
  • Folder tokens use s and degC for clarity.

Master Flats

  • Created under replicated output tree unless inline mode is selected.
  • Format controlled by Output Format setting (XISF or FITS).

9. UI Behavior and Safety Rules

  • Configuration values and selected roots are persisted in AppData.
  • Run Matching is manual and non-destructive.
  • Scan and processing expose Abort; long loops are cancellation-aware.
  • Matching table supports sorting, grouped expansion, include filtering, and per-group dark override.
